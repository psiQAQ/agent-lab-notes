# 文档编写偏好

本仓库后续新增或更新 `skills`、`plugins`、Agent 工具链、AI 辅助项目说明文档时，默认遵循下面这些写法偏好。

## 1. 总体风格

- 以中文写作，专业、直接、少废话。
- 优先写成“可直接使用的说明文档”，不要写成长篇项目介绍或宣传稿。
- 先给读者最重要的信息：这是什么、适合谁、怎么装、怎么用。
- 避免大段背景铺垫；背景只写到能帮助理解为止。
- 如果用户没有要求，不要展开太多底层实现、内部架构或编程细节。

## 2. 插件 / skill / Agent 项目文档

这类文档优先采用下面的顺序：

1. 开头一句话说明项目是什么
2. 如果有多个仓库，先说明分别对应哪个 Agent / 平台
3. 给出项目地址
4. 说明它如何影响 Agent 的工作方式
5. 分平台写安装步骤
6. 给少量高质量通用提示词

默认不要：

- 写很长的能力总览
- 堆太多命令、模式、开关
- 把 README 的所有内容都搬过来
- 过度偏向某一个 Agent，除非用户明确要求

## 3. 多仓库项目的写法

如果一个项目同时维护多个仓库：

- 先做中立总览
- 明确每个仓库分别服务哪个平台或 Agent
- 不把它们写成“主次争夺”关系，除非官方文档明确这样定义
- 对安装、入口、命令分别写清楚，不混在一起

如果用户明确说“不要偏向某一个 Agent”，就：

- 不用某一端的术语主导全文
- 不把某一端当默认读者
- 用“分别安装、分别调用、通用提示词”这种结构

## 4. 提示词示例的写法

- 数量少，质量高。
- 提示词优先覆盖最常见任务，不追求面面俱到。
- 一条提示词只解决一个明确阶段的问题。
- 优先写“收敛问题、规划结构、自查审稿、回复意见”这类高价值任务。
- 不要堆很多相似提示词。
- 默认强调先明确阶段，再要求输出。

## 5. graphify 类项目的写法

如果是像 `graphify` 这种工具型项目：

- 尽量保留原有教程框架，除非用户要求重构
- 重点核对旧仓库名、旧平台列表、旧安装命令是否过时
- 统一改成当前官方口径
- 明确区分：系统命令、对话命令、平台常驻安装
- 对 `Codex`、`Claude Code`、`OpenCode` 这类平台分别写清执行位置

## 6. academic / research 类项目的写法

- 不写成大而泛的研究方法论文式介绍
- 聚焦：这是什么、给谁用、怎么装、怎么驱动 Agent
- 如果用户要求精简，就只保留：
  - 项目介绍
  - 仓库对应关系
  - 安装
  - 通用提示词

## 7. 更新旧文档时

- 优先核对官方仓库当前 README / CHANGELOG / 安装文档
- 如果旧文档时间较早，要显式修正：
  - 仓库地址
  - 平台支持列表
  - 当前推荐安装命令
  - 当前推荐入口命令
- 能保留原框架时尽量保留，但内容应更新到当前口径

## 8. Git 相关

- 这类文档更新如果已经可用，默认可以纳入一次正常提交。
- 提交时优先只带当前任务相关文件。
- 如果用户明确要求“上传”，则在提交后推送远端。
